# Final Pre-Launch Audit - Main Site Only

Audit date: 2026-04-29  
Scope: `Main` only. The `learn` app is intentionally excluded per latest instruction.

## 1. Executive Summary

Overall launch readiness score: 8/10

Recommendation: Ready to publish after final env/config confirmation.

The main Bad Decisions Studio site is close to production-ready. It is a static site generated from templates/sections by `build.js`, has strong core metadata, clean internal routing, no checked-in private env files, production cache/security headers in `vercel.json`, and no missing local assets or missing image alt text in the built pages checked.

Top 5 highest-priority issues:

1. Confirm Vercel env vars for podcast cron: `CRON_SECRET` and `VERCEL_DEPLOY_HOOK_URL` are required for `/api/cron/podcast-refresh`.
2. CSP has been hardened and verified: blanket inline script/style allowances were removed and all 12 current JSON-LD blocks are covered by explicit hashes.
3. Homepage preload pressure was reduced by removing non-critical font preloads and the hero-video preload.
4. Lighthouse is strong on desktop but mobile performance still needs attention: mobile Performance 79, desktop Performance 99.
5. Newsletter forms post directly to Klaviyo from the browser; public IDs are expected, but privacy/consent language should get final legal/product review.

## 2. Critical Issues

### Issue 1

Severity: High  
Category: Security / Production readiness  
File/location: `Main/api/cron/podcast-refresh.js:12-24`, `Main/vercel.json:4-8`  
Problem: Daily cron depends on `CRON_SECRET` and `VERCEL_DEPLOY_HOOK_URL`. If either is missing, the endpoint correctly fails.  
Risk: Podcast episode refresh will not run after publish.  
Recommended fix: Configure both values in Vercel production env. Keep values private.  
Status: Reported only.

### Issue 2

Severity: Medium  
Category: Security  
File/location: `Main/vercel.json:31`  
Problem: CSP used `script-src 'self' 'unsafe-inline'` and `style-src 'self' 'unsafe-inline'`.  
Risk: Inline execution allowances reduced CSP protection if any injection bug was introduced later.  
Recommended fix: Replace broad inline allowances with explicit hashes for the current JSON-LD blocks and remove inline style allowance.  
Status: Fixed. `Main/vercel.json` now uses explicit SHA-256 hashes for the 12 current inline JSON-LD blocks, removes `'unsafe-inline'`, adds `object-src 'none'`, and enables `upgrade-insecure-requests`.

### Issue 3

Severity: Medium  
Category: Performance  
File/location: `Main/templates/index.html:28-35`  
Problem: Homepage preloaded six font files and the hero video.  
Risk: More high-priority bytes on first load, especially on mobile.  
Recommended fix: Keep only truly critical font/video preloads, measure LCP on preview, and lazy/defer anything below the fold.  
Status: Fixed. `Main/templates/index.html` now preloads four critical fonts instead of six and no longer preloads `hero-video.webm`; `Main/index.html` was regenerated by `npm run build`.

### Issue 4

Severity: Low  
Category: Security / Privacy  
File/location: `Main/js/newsletter.js:6-105`, `Main/sections/newsletter.html`  
Problem: Newsletter sends name/email directly to Klaviyo from the browser. Public Klaviyo identifiers are present in client code.  
Risk: Public IDs are not secrets, but form submission transmits contact data to a third party.  
Recommended fix: Confirm privacy copy, consent requirements, and Klaviyo list/source configuration before launch.  
Status: Reported only. I did not submit the form because that would transmit contact data.

### Issue 5

Severity: Low  
Category: Code  
File/location: `Main/dev-server.js:5`, `Main/dev-server.js:50`  
Problem: Local server imported and used Node's legacy `url.parse()`, which emitted a deprecation warning under the local Node version.  
Risk: Dev-only warning/noise; not a production Vercel issue.  
Recommended fix: Replace with WHATWG `new URL(reqUrl, base)`.  
Status: Fixed. `Main/dev-server.js` now uses `new URL(reqUrl, 'http://localhost')` and returns 404 for malformed URL/path decoding instead of throwing.

## 4. Performance Findings

Main bottlenecks:

- Mobile Lighthouse reports Performance 79, with LCP at 4.9s. Desktop is excellent at Performance 99, LCP 0.9s.
- The largest payloads on the homepage are video assets: `Clients.mp4` (~2.14 MB), `hero-video.mp4` (~2.05 MB), `education-reel.mp4` (~1.25 MB), and `podcast-reel.mp4` (~902 KB).
- Lighthouse identified the lazy-loaded `80 Level` client logo as the LCP element on mobile, which likely means above-the-fold animation/scroll timing is influencing LCP attribution.
- CSS has room for later reduction: estimated unused CSS is ~139 KiB on mobile and ~160 KiB on desktop.
- Homepage preload pressure has been reduced; post-fix Lighthouse confirms desktop is very strong, while mobile still needs asset/LCP work.
- Several videos are used throughout sections; most use metadata preload or lazy behavior, which is good, but total homepage byte weight is still ~7.49 MB.
- Static assets are cached with long-lived immutable headers in `vercel.json`.

Fixes implemented:

- Hardened CSP in `Main/vercel.json` by removing blanket `'unsafe-inline'` allowances, adding hashes for inline JSON-LD, and adding stricter directives.
- Verified CSP hash coverage after the change: 12 inline scripts, 12 unique hashes, 12 allowed hashes, 0 missing hashes.
- Removed non-critical homepage preloads in `Main/templates/index.html` and rebuilt `Main/index.html`.
- Removed lazy loading from the first visible homepage trusted-logo marquee set so Lighthouse does not wait on lazy LCP logo discovery.
- Deferred non-hero homepage video loading by switching pillar/about videos to `preload="none"` and updating `Main/js/main.js` to start non-hero `data-autoplay` videos only near viewport entry.
- Added a lightweight mobile hero video source, `Main/assets/video/hero-video-480.mp4`, and wired it for screens up to 768px before the desktop video sources.
- Added `Main/favicon.ico` for legacy browser/crawler compatibility.
- Added `npm test` smoke coverage for generated pages, JSON-LD parsing, unresolved include/token checks, and CSS/JS cache-busting.
- Added a build-time assertion that fails if any `data-include` or `{{token}}` remains unresolved after page generation.
- Declared the Node runtime requirement with `engines.node >=18`.
- Replaced the dev server's legacy URL parser in `Main/dev-server.js`.
- Updated the dev server to stream files with `fs.createReadStream()` instead of reading full assets into memory.

Recommended later:

- Run Lighthouse again on a Vercel preview, because local static serving does not perfectly match production CDN behavior.
- Improve mobile LCP by reviewing the above-the-fold logo marquee and hero/video loading order.
- Consider lighter mobile video variants or poster-first loading for heavy video sections.
- Split or reduce unused CSS after launch if performance budget requires it.
